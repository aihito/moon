# 游戏服网关 · 多节点扩展架构

本文描述**多台 Moon 节点**水平扩展时的架构：游戏服如何挂到不同节点、匹配策略（本节点 / 全局）、以及与「单节点多 bridge」的组合方式。单节点基础接入见 [game_server_gate_single_node.md](game_server_gate_single_node.md)；单节点内多游戏服（多 bridge）也在该文档 §8.1。

---

## 1. 适用场景

- 单机扩展能力到顶，需要**多机部署**。
- 按**区服 / 大区隔离**（如一区一 Moon 节点）。
- 游戏服数量多且希望分散到多台 Moon，做负载均衡与容灾。

---

## 2. 多 Moon 节点（方案 B）

**思路**：部署**多台 Moon 节点**，每台仍是 bridge + center + room 的网关架构；每台有独立 listen 端口。**游戏服按策略连到某一台 Moon**（如 zone_id 取模、game_server_id 哈希、或配置表），该节点内可以是单 bridge（一节点一游戏服）或多 bridge（一节点多游戏服，见 gate 文档 §8.1）。

```
┌─────────────┐         ┌─────────────────────────────────────┐
│  游戏服 1   │────────►│  Moon 节点 1（zone_1 / 大区 A）       │
│  (zone_1)   │         │  bridge(s) + center + room           │
└─────────────┘         └─────────────────────────────────────┘
┌─────────────┐         ┌─────────────────────────────────────┐
│  游戏服 2   │────────►│  Moon 节点 2（zone_2 / 大区 B）       │
│  (zone_2)   │         │  bridge(s) + center + room           │
└─────────────┘         └─────────────────────────────────────┘
       ...                        ...
```

### 2.1 游戏服连到哪台节点

可选策略（可组合）：

| 策略 | 说明 |
|------|------|
| **按区服 / 大区** | 配置表或 DNS：zone_id / 大区 id → Moon 节点地址:端口；游戏服启动时查表连接。 |
| **按 game_server_id 哈希** | `node_id = hash(game_server_id) % node_count`，保证同一游戏服总是连同一节点。 |
| **统一网关 / 负载均衡** | 前层再挂 L4/L7 或自研网关，按请求特征（如 zone、player_id 前缀）转发到对应 Moon 节点。 |

### 2.2 匹配策略

| 策略 | 说明 | 适用 |
|------|------|------|
| **仅本节点匹配** | 每节点 center 只对本节点内玩家做 match_queue / 进房；跨节点不匹配。 | 按区服隔离、实现简单、无需跨节点通信。 |
| **全局匹配 + 房间落某节点** | 单独做「全局匹配服务」（可独立进程或挂在某 Moon），凑齐人后选定一个 Moon 节点创建 room，并通知双方游戏服「该房间在节点 X」；游戏服把后续房间内上行发往节点 X（或由网关按 room_id 路由）。 | 跨区服匹配、多节点负载均衡；需额外路由与状态同步。 |

**建议**：先做**仅本节点匹配**，节点间无依赖；若后续要跨区匹配，再引入全局匹配服务与「房间所在节点」的路由。

---

## 3. 多节点 + 每节点多 bridge（方案 A+B）

在方案 B 基础上，每台 Moon 节点内部采用「一游戏服一 bridge」：每 `accept` 一个连接就 `new_service(bridge)` 并 `set_fd`，center 维护 `player_id → bridge_id`，下行按 bridge 转发。这样既有多机水平扩展，又支持单节点多游戏服。

- **节点间**：游戏服按 zone_id / 哈希 / 配置连到某节点。
- **节点内**：多 bridge + 唯一 center，center 维护 `player_bridge` 与 `player_room`，详见 [game_server_gate_single_node.md](game_server_gate_single_node.md) §8.1。
- **匹配**：可先仅本节点；后续再接全局匹配与跨节点房间路由。

---

## 4. 全局匹配与房间路由（可选）

若要做**跨节点匹配**，可单独部署一个**全局匹配服务**（或挂在某 Moon 上）：

1. 各节点 center 将「待匹配玩家」上报全局匹配（或玩家直接连全局匹配）。
2. 全局匹配凑齐一局后，选定一个 Moon 节点（如负载最低、或按 zone 策略）创建 room，并返回「房间所在节点 + room_id」给双方。
3. 游戏服侧：收到「房间在节点 X」后，将后续该房间的上行（room_action 等）发往**节点 X** 的地址（或由统一网关根据 room_id 路由到节点 X）。
4. 节点 X 的 bridge 收到上行后，按既有逻辑查 `player_room` 转 room；下行仍由该节点 room → bridge → 游戏服。

跨节点时需约定：player_id 全局唯一，或 (node_id, player_id) 唯一；房间创建与销毁由目标节点负责，全局匹配只做「选节点 + 通知」。

---

## 5. 小结与选型

| 场景 | 方案 | 文档位置 |
|------|------|----------|
| 单机、单/多游戏服 | 单节点单 bridge 或单节点多 bridge | [game_server_gate_single_node.md](game_server_gate_single_node.md) 正文及 §8.1 |
| 多机、按区服隔离 | 多 Moon 节点，仅本节点匹配 | 本文 §2 |
| 多机 + 单节点多游戏服 | 多节点 + 每节点多 bridge | 本文 §3 + gate 文档 §8.1 |
| 跨区服匹配 | 多节点 + 全局匹配 + 房间落某节点 | 本文 §2.2、§4 |

实现顺序建议：先单节点多 bridge 跑通（gate 文档 §8.1），再按需加多节点与「仅本节点匹配」（本文 §2）；最后若需要再上全局匹配与跨节点房间（本文 §4）。
