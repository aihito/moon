# 游戏服网关 · 多节点扩展架构

本文描述**多台 Moon 节点**水平扩展时的架构：游戏服如何挂到不同节点、匹配策略（本节点 / 全局）、以及与「单节点多 bridge」的组合方式。单节点基础接入见 [game_server_gate_single_node.md](game_server_gate_single_node.md)；单节点内多游戏服与 bridge 数量/形态见该文档 §8。

---

## 1. 架构总览

### 1.1 概念与角色

| 概念 | 说明 |
|------|------|
| **Moon 节点** | 一台（或一组）Moon 进程，对外一个 listen 端口；内部是 bridge(s) + center + room 的网关架构。多节点 = 多台机器/多组进程，彼此独立，无内置跨节点通信。 |
| **游戏服** | 已有业务服；与**某一台** Moon 节点建立**一条** TCP 连接，负责上下行转发。玩家在游戏服登录，扩展玩法（匹配/房间/战斗）的消息经游戏服转发到 Moon。 |
| **bridge** | Moon 内持有与游戏服 fd 的服务；维护本连接上玩家的 `player_id → room_id` 路由；上行解包转 center/room，下行收 center/room 的 forward 写 fd。单节点可多 bridge（一游戏服一 bridge）。 |
| **center** | Moon 内唯一；匹配队列、创建/销毁房间、维护 `player_id → bridge_id` 与 `player_id → room_id`。**每节点一个 center**，默认只处理本节点玩家。 |
| **room** | 房间服务；房间内逻辑（如战斗）。由 center 创建，下行经 bridge 回游戏服。 |

多节点时：**节点间**通过「游戏服连到哪台节点」和（可选）「全局匹配 + 房间落某节点」来分工；**节点内**与单节点方案一致（bridge + center + room）。

### 1.2 多节点拓扑示意

```
                         ┌─────────────────────────────────────────────────────────┐
                         │                    游戏服层（已有）                        │

                         │   │(zone_1)  │  │(zone_2)  │  │(zone_1)  │  │(zone_K)  │  │

- 每台 Moon 节点：独立 listen 端口，内部 bridge(s) + 一个 center + 多个 room。

## 2. 适用场景

| 场景 | 说明 |
|------|------|
| **单机扩展能力到顶** | 需要多机部署，水平扩展 Moon 网关。 |
| **按区服 / 大区隔离** | 一区一 Moon 节点，节点间无状态共享，实现简单。 |
| **多游戏服负载均衡与容灾** | 游戏服数量多，分散到多台 Moon，避免单点。 |

## 3. 游戏服接入策略：连到哪台节点

多节点下，每个游戏服需要确定连到**哪一台** Moon 节点。可选策略（可组合使用）：

```
┌────────────────────────────────────────────────────────────────────────────┐
│  策略一：按区服 / 大区（配置表或 DNS）                                        │
│  zone_id / 大区 id → Moon 节点地址:端口；游戏服启动时查表连接。                 │
│                                                                             │
│    zone_1 ──► node_1:7001    zone_2 ──► node_2:7002    zone_3 ──► node_3:7003  │
└────────────────────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────────────────────┐
│  策略二：按 game_server_id 哈希                                               │
│  node_id = hash(game_server_id) % node_count                                 │
│  同一游戏服总是连同一节点，便于会话保持。                                       │
└────────────────────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────────────────────┐
│  策略三：统一网关 / 负载均衡（L4/L7 或自研网关）                               │
│  前层按请求特征（如 zone、player_id 前缀）转发到对应 Moon 节点。                │
│                                                                             │
│    [ 游戏服 ] ──► [ 网关 ] ──┬──► node_1  (zone_1 / 前缀 A)                   │
│                             ├──► node_2  (zone_2 / 前缀 B)                   │
│                             └──► node_3  (zone_3 / 前缀 C)                   │
└────────────────────────────────────────────────────────────────────────────┘
```

| 策略 | 说明 |
|------|------|
| **按区服 / 大区** | 配置表或 DNS：zone_id / 大区 id → Moon 节点地址:端口；游戏服启动时查表连接。 |
| **按 game_server_id 哈希** | `node_id = hash(game_server_id) % node_count`，保证同一游戏服总是连同一节点。 |
| **统一网关 / 负载均衡** | 前层再挂 L4/L7 或自研网关，按请求特征（如 zone、player_id 前缀）转发到对应 Moon 节点。 |

## 4. 匹配策略（本节点 vs 全局）

### 4.1 仅本节点匹配（推荐先实现）

每台 Moon 的 **center 只对本节点内的玩家**做匹配与进房；跨节点不匹配，节点间无依赖。

```
  Moon 节点 1                              Moon 节点 2
  ┌─────────────────────────┐              ┌─────────────────────────┐
  │ 游戏服 A ──► bridge_A    │              │ 游戏服 B ──► bridge_B    │
  │     玩家 p1, p2          │              │     玩家 p3, p4          │
  │            │             │              │            │             │
  │            ▼             │              │            ▼             │
  │     center（仅看本节点）   │              │     center（仅看本节点）   │
  │     match_queue: p1,p2   │              │     match_queue: p3,p4   │
  │            │             │              │            │             │
  │            ▼             │              │            ▼             │
  │     room_1 (p1, p2)      │              │     room_1 (p3, p4)      │
  └─────────────────────────┘              └─────────────────────────┘
        节点内匹配，互不通信
```

| 项 | 说明 |
|------|------|
| **优点** | 按区服隔离、实现简单、无需跨节点通信；每节点独立扩容。 |
| **适用** | 按区服/大区隔离、同区玩家匹配即可。 |

### 4.2 全局匹配 + 房间落某节点（可选扩展）

需要**跨区服匹配**时，可引入**全局匹配服务**：凑齐一局后选定**一个** Moon 节点创建房间，双方游戏服将后续房间消息发往该节点。

```
  ┌─────────────┐     ┌─────────────┐     ┌─────────────────────────────────────────┐
  │ 游戏服 A    │     │ 游戏服 B    │     │ 全局匹配服务（独立进程或挂某 Moon）         │
  │ 节点 1      │     │ 节点 2      │     │ 收各节点/玩家的待匹配信息，凑齐后选节点建房   │
  └──────┬──────┘     └──────┬──────┘     └──────────────────┬──────────────────────┘
         │                   │                               │
         │ 上报待匹配 / 拉取  │                               │ 返回 (node_id, room_id)
         └───────────────────┴───────────────────────────────┘
                                         │
         ┌───────────────────────────────┼───────────────────────────────┐
         ▼                               ▼                               ▼
  ┌─────────────┐                 ┌─────────────┐                 ┌─────────────┐
  │ Moon 节点 1 │                 │ Moon 节点 2 │                 │ Moon 节点 3 │
  │ center      │                 │ center      │                 │ center      │
  │ room ...    │                 │ room ...    │  创建 room 后    │ room_new    │
  └─────────────┘                 └─────────────┘  房间消息发往 ──►└─────────────┘
                                                                  节点 3
```

**流程简述**：

1. 各节点 center 将「待匹配玩家」上报全局匹配（或玩家/游戏服直接连全局匹配）。
2. 全局匹配凑齐一局后，选定一个 Moon 节点（如负载最低、或按 zone 策略）在该节点创建 room，并返回「房间所在节点 + room_id」给双方。
3. 游戏服侧：收到「房间在节点 X」后，将后续该房间的上行（room_action 等）发往**节点 X** 的地址；或由统一网关根据 room_id 路由到节点 X。
4. 节点 X 的 bridge 收到上行后，按既有逻辑查 `player_room` 转 room；下行仍由该节点 room → bridge → 游戏服。

**约定**：player_id 全局唯一，或 (node_id, player_id) 唯一；房间创建与销毁由目标节点负责，全局匹配只做「选节点 + 通知」。

## 5. 方案组合：多节点 + 每节点多 bridge

在「多台 Moon 节点」基础上，每台节点内部可以采用**一游戏服一 bridge**：每 `accept` 一个连接就 `new_service(bridge)` 并 `set_fd`，本节点内 **center 唯一**，维护 `player_id → bridge_id` 与 `player_id → room_id`，下行按 bridge 转发。

```
  节点间                                    节点内（单节点多 bridge）
  游戏服 ──按策略──► 某 Moon 节点           同 [game_server_gate_single_node.md](game_server_gate_single_node.md) §8.2

  ┌─────────────┐    ┌─────────────────────────────────────────────────────────────┐
  │ 游戏服 A    │───►│  Moon 节点 1                                                  │
  │ 游戏服 B    │───►│  ┌─────────┐ ┌─────────┐ ┌─────────┐                          │
  │ 游戏服 C    │───►│  │bridge_A │ │bridge_B │ │bridge_C │  ◄── 一连接一 bridge      │
  └─────────────┘    │  └────┬────┘ └────┬────┘ └────┬────┘                          │
                    │       └──────────┼──────────┘                                 │
                    │                  ▼                                            │
                    │  ┌──────────────────────────────────────────────────────────┐ │
                    │  │ center（本节点唯一）                                       │ │
                    │  │ player_id → bridge_id ； player_id → room_id              │ │
                    │  └──────────────────────────────────────────────────────────┘ │
                    └─────────────────────────────────────────────────────────────┘
```

| 层级 | 说明 |
|------|------|
| **节点间** | 游戏服按 zone_id / 哈希 / 配置连到某节点。 |
| **节点内** | 多 bridge + 唯一 center；center 维护 `player_bridge` 与 `player_room`，详见 [game_server_gate_single_node.md](game_server_gate_single_node.md) §8.2。 |
| **匹配** | 可先仅本节点；后续再接全局匹配与跨节点房间路由。 |

## 6. 场景：某地区所有服、所有玩家共同参与匹配

### 6.1 需求与目标

- **需求**：同一地区内的**所有游戏服**、**所有玩家**进入**同一个匹配池**，任意两名玩家只要在该地区即可被匹配到一局。
- **目标**：架构清晰、易实现、易运维；可选的扩展路径（地区内多节点）。

### 6.2 架构选型与原因

| 方案 | 思路 | 适用 | 推荐度 |
|------|------|------|--------|
| **一地区一 Moon 节点** | 该地区只部署**一个** Moon 节点，区内**所有游戏服**都连到这同一节点（多 bridge）；一个 center、一个 match_queue，天然全区共享匹配池。 | 单节点能承载该地区匹配+房间+战斗的负载时。 | **优先** |
| **一地区多 Moon 节点 + 地区匹配服务** | 地区内多台 Moon 节点 + 独立的「地区匹配服务」汇聚各节点待匹配玩家，凑齐后选定一个节点建房；游戏服按房间所在节点转发后续消息。 | 单节点容量或可用性不足，需要地区内多机水平扩展时。 | 备选 |

**推荐「一地区一 Moon 节点」的原因**：

1. **实现简单**：无需跨节点协议、无需额外匹配服务，就是 [game_server_gate_single_node.md](game_server_gate_single_node.md) §8.2 的「单节点多 bridge」；center 唯一，match_queue 自然包含所有连上来的玩家。
2. **语义自然**：一个 center = 一个匹配池 = 全区服全玩家，无分区、无路由歧义。
3. **延迟一致**：匹配与房间都在同一节点，无跨节点转发。
4. **运维清晰**：按地区扩容即按地区增加 Moon 节点数，地区内仍是一节点一池。

当单节点（单机或单进程组）无法承受该地区流量时，再考虑「一地区多节点 + 地区匹配服务」（见 §6.5）。

### 6.3 方案一：一地区一 Moon 节点（推荐）— 架构详解

**拓扑**：地区内 N 台游戏服 → 同一 Moon 节点（N 个 bridge）→ 一个 center → 多个 room。

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│  某地区（如华东区）                                                                        │
│                                                                                          │
│   ┌─────────────┐ ┌─────────────┐ ┌─────────────┐         ┌─────────────┐                │
│   │ 游戏服 1    │ │ 游戏服 2    │ │ 游戏服 3    │   ...   │ 游戏服 K    │                │
│   │ 玩家 a,b,c  │ │ 玩家 d,e   │ │ 玩家 f,g,h  │         │ 玩家 ...    │                │
│   └──────┬──────┘ └──────┬──────┘ └──────┬──────┘         └──────┬──────┘                │
│          │                │                │                     │                       │
│          │ 全部连到同一 Moon 节点（按地区配置或 DNS）                                        │
│          ▼                ▼                ▼                     ▼                       │
│   ┌─────────────────────────────────────────────────────────────────────────────────────┐│
│   │  Moon 节点（本地区唯一）  listen :7001                                                 ││
│   │  ┌─────────┐ ┌─────────┐ ┌─────────┐     ┌─────────┐                                 ││
│   │  │bridge_1 │ │bridge_2 │ │bridge_3 │ ... │bridge_K │  一游戏服一 bridge，一连接一 fd  ││
│   │  └────┬────┘ └────┬────┘ └────┬────┘     └────┬────┘                                 ││
│   │       └───────────┴───────────┴───────────────┘                                       ││
│   │                              │                                                        ││
│   │                              ▼                                                        ││
│   │  ┌──────────────────────────────────────────────────────────────────────────────────┐ ││
│   │  │ center（本节点唯一）                                                              │ ││
│   │  │ match_queue：全区所有 join_match 的玩家（来自任意游戏服）                           │ ││
│   │  │ player_id → bridge_id ； player_id → room_id                                      │ ││
│   │  └──────────────────────────────────────────────────────────────────────────────────┘ ││
│   │  ┌────────┐ ┌────────┐     ┌────────┐                                                ││
│   │  │ room_1 │ │ room_2 │ ... │ room_M │  满人即 new_service(room)，无服/节点边界        │ ││
│   │  └────────┘ └────────┘     └────────┘                                                ││
│   └─────────────────────────────────────────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────────────────────────────────────────┘
```

**数据流要点**：

- **上行**：任意游戏服收到玩家「参与匹配」→ 经本服与 Moon 的连接发 `join_match` → 对应 bridge → center；center 将玩家加入**同一个** match_queue，不区分来自哪台游戏服。
- **匹配**：center 凑齐一局后 `new_service(room)`，通知各玩家对应的 bridge 更新 `player_room`；这些玩家可能来自不同游戏服（不同 bridge），center 通过 `player_bridge` 分别下发。
- **房间内**：房间内上行经各自游戏服 → 各自 bridge → 同一 room；下行 room → bridge（按 player_id 可能多个 bridge）→ 各游戏服 → 各自玩家。

**关键约定**：

- **player_id 地区内唯一**：同一地区内所有游戏服对玩家的 id 分配需保证唯一（如全局 UUID、或 game_server_id + 服内 id 拼接），避免不同服两个玩家同 id 冲突。
- **地区与节点映射**：配置表或 DNS：`region_id / 地区名 → Moon 节点地址:端口`；该地区内所有游戏服启动时查同一地址连同一节点。

### 6.4 方案一实现方案（步骤清单）

| 步骤 | 内容 |
|------|------|
| **1. Moon 侧** | 按 [game_server_gate_single_node.md](game_server_gate_single_node.md) 实现单节点网关；多游戏服时按 §8.2 做「一连接一 bridge」（或 §8.1/§8.3 单 bridge / 按维度 bridge），center 维护 `player_bridge` 与 `player_room`，匹配逻辑不区分 bridge 来源。 |
| **2. 配置** | 地区 → Moon 节点：`region_id → "host:port"`；游戏服启动时根据本服所在地区读取配置，建立到该 Moon 的唯一连接。 |
| **3. 游戏服侧** | 与 Moon 建立一条 TCP；上行带 `player_id`（地区内唯一）、`cmd`（如 `join_match`、`room_action`）；下行根据 Moon 下发的 `player_id`(s) 转发给本服对应玩家。 |
| **4. player_id 唯一** | 约定地区内唯一：例如 `player_id = tostring(game_server_id) .. "_" .. tostring(in_game_uid)`，或由中心 ID 服务分配。 |
| **5. 可选** | 首包或心跳中游戏服可带 `game_server_id` / `region_id`，便于 Moon 打日志或监控；匹配逻辑无需区分。 |

无需额外「匹配服务」或跨节点协议；扩容时按地区增加 Moon 节点数，每个地区仍是一节点一匹配池。

### 6.5 方案二：一地区多 Moon 节点 + 地区匹配服务（可选）

当**单节点无法承受该地区负载**或需要**地区内多机容灾**时，可采用：

- **拓扑**：地区内多台 Moon 节点（每台部分游戏服连接）+ 一个**地区级匹配服务**（独立进程或挂在某 Moon 上，仅负责本地区）。
- **流程**：各节点 center 将本节点待匹配玩家上报地区匹配服务；地区匹配服务凑齐一局后，选定本地区内一个 Moon 节点创建 room，并返回「房间所在节点 + room_id」；游戏服将后续该房间的上行发往该节点（或经网关按 room_id 路由）。
- **与 §4.2 的关系**：逻辑同多节点文档 §4.2「全局匹配 + 房间落某节点」，只是「全局」限定为本地区，player_id 只需地区内唯一；跨地区匹配可再上层做「跨地区匹配服务」。

实现要点：地区匹配服务需与各 Moon 节点通信（或各 center 主动上报 + 拉取）；房间创建由被选中的节点执行；游戏服或网关需支持按 `(region, node_id, room_id)` 路由上行到对应节点。

### 6.6 方案三：匹配节点与房间节点分离（Center + Room，Gate 非必须）

「一个节点负责匹配，其他节点负责房间战斗」是可行的；且 **Gate 不是必须的**：游戏服匹配阶段可直接连 **Center**，由 Center 持连接并下发 match_ok，房间阶段游戏服直连 **Room**，架构只需 **Center + Room** 两类节点。

#### 为何 Gate 可省略

- 匹配阶段只需要：收 join_match、做匹配、建房后下发 match_ok(room_addr, room_id)。
- 若游戏服**直连 Center**，由 Center 自己持连接（listen + 一游戏服一 fd / 一连接一 bridge），则 Center 即可收 join_match、做匹配、RPC 建房后**直接在同一连接上**下发 match_ok，无需经过单独的 Gate 节点。
- 房间阶段仍是游戏服按 match_ok 中的 room_addr 直连 Room 节点。因此**两角色足够**：Center（匹配 + 匹配阶段连接）、Room（房间 + 房间阶段连接）。

可选：若希望匹配层与「持连接」层解耦，可在中间加一层 Gate（游戏服 → Gate → Center，Center 通知 Gate 下发 match_ok），但多数场景不必多这一跳。

#### 角色与拓扑（无 Gate）

- **Center 节点**：**匹配 + 匹配阶段连接**。对游戏服 listen，一游戏服一连接（或一连接一 bridge）；收 join_match 入 match_queue，满人后 RPC 到某 Room 节点建房，得到 `(room_addr, room_id)`，**经已有连接**向对应游戏服下发 match_ok(room_addr, room_id)。不参与房间阶段流量。
- **Room 节点**：**房间 + 房间阶段连接**。对外 listen 供游戏服「加入房间」；接受游戏服连接后首包绑定 room_id + 本服玩家列表；房间内 room_action / forward 均走该连接。
- **游戏服**：匹配阶段连 **Center**，发 join_match，收 match_ok(room_addr, room_id)；房间阶段连 **room_addr**，发 attach_room 后收发房间消息。两条连接（Center、Room），无 Gate。

#### 架构示意（Center + Room，无 Gate）

```
  匹配阶段：游戏服 ──────────► Center（持连接 + 匹配 + 下发 match_ok）
  房间阶段：游戏服 ◄────────────► Room 节点（直连）

                         ┌─────────────────────────────────────────────────────────────────┐
                         │  游戏服（全区）                                                   │
                         │  ┌─────────┐ ┌─────────┐ ┌─────────┐                            │

                         │  └────┬────┘ └────┬────┘ └────┬────┘                            │

                                 │          │          │
     匹配阶段：join_match / match_ok（直连 Center）     │
                                 ▼          ▼          ▼
  ┌─────────────────────────────────────────────────────────────────────────────────────────┐

  │                       │                                                                  │
  │                       ▼                                                                  │
  │  ┌─────────────────────────────────────────────────────────────────────────────────────┐ │

  │  │ room/room   │    │ room/room   │    │ room/room   │  游戏服收到 match_ok 后直连本节点   │

#### 数据流

1. **join_match**：游戏服 → **Center**（经匹配阶段连接）；Center 将玩家加入 match_queue。
2. **凑齐一局**：Center 选一个 Room 节点，RPC「创建房间(玩家列表)」，得到 `room_addr + room_id`。
3. **match_ok**：Center **经同一连接**向各游戏服下发：匹配成功，`room_addr`，`room_id`。此后该局流量 Center 不再参与。
4. **加入房间连接**：各游戏服根据 `room_addr` 主动连 Room 节点，发送「attach_room room_id，本服玩家 [p1, p2, ...]」；Room 将连接与 room_id、玩家列表绑定。
5. **room_action**：游戏服 → 到 Room 的连接 → 对应 room；room 处理逻辑。
6. **下行 forward**：room → 写回该房间对应的游戏服连接 → 游戏服按 player_id 转发给玩家。

#### 实现要点

| 项 | 说明 |
|------|------|
| **Center 持连接** | Center 需 listen 游戏服端口，一游戏服一连接（或 bridge）；维护 player_id/连接与 match_queue 的对应，满人后在同一连接上发 match_ok，无需 Gate。 |
| **Room 节点对游戏服的 listen** | Room 除「Center 建房 RPC」外，需单独端口供游戏服连接；首包约定为「attach_room room_id, player_ids[]」，之后为该房间的 room_action / forward。 |
| **同一房间多游戏服** | 一局多台游戏服时，每台各自连 Room，都发「attach_room room_id, 本服玩家」；Room 将多连接绑定到同一 room_id，广播时按 player_id 找连接写回。 |
| **room_addr** | match_ok 中 room_addr 可为 Room 节点 host:port，或统一网关 + room_id 路由。 |
| **可选 Gate** | 若要把「持连接」与「匹配逻辑」拆成两进程，可加 Gate：游戏服连 Gate，Gate 转 Center，Center 通知 Gate 下发 match_ok；多数场景不必。 |

#### 与「一地区一节点全功能」的对比

| 维度 | 一地区一节点（Gate+Match+Room 同节点）§6.3 | 匹配与房间分离（Center + Room，无 Gate）§6.6 |
|------|--------------------------------------------|---------------------------------------------|
| **实现复杂度** | 低，无跨节点 | 中：Center 与 Room 建房 RPC；Room 接受游戏服连接；游戏服两条连接（Center / Room） |
| **节点角色** | 单节点全功能 | **仅 Center + Room**，无 Gate |
| **扩展方式** | 按地区加节点 | Center 单点或主备；**只扩 Room 节点**即可扛房间流量 |
| **延迟** | 房间同进程 | 房间阶段游戏服↔Room 直连，匹配阶段游戏服↔Center 直连 |

**结论**：Gate 不是必须的；游戏服直连 Center 做匹配、直连 Room 做房间，**Center + Room** 两类节点即可完成「匹配节点与房间节点分离」，架构更简单。

可运行示例：`example/guess_gate_multinode_center_room/`（Center 节点单 bridge 多连接 + Room 节点，猜数字玩法）。

## 7. 小结与选型

| 场景 | 方案 | 文档位置 |
|------|------|----------|
| 单机、单/多游戏服 | 单节点单 bridge 或单节点多 bridge | [game_server_gate_single_node.md](game_server_gate_single_node.md) 正文及 §8 |
| **某地区所有服所有玩家共同匹配** | **一地区一 Moon 节点**（区内所有游戏服连同一节点，多 bridge） | 本文 §6.3、§6.4 |
| **同地区、匹配轻/战斗重，希望多机只扩房间** | **Center + Room**（游戏服直连 Center 匹配、直连 Room 房间，**Gate 非必须**） | 本文 §6.6 |
| 多机、按区服隔离 | 多 Moon 节点，仅本节点匹配 | 本文 §3、§4.1 |
| 多机 + 单节点多游戏服 | 多节点 + 每节点多 bridge | 本文 §5 + gate 文档 §8.2 |
| 跨区服匹配 / 地区内多节点匹配 | 多节点 + 全局/地区匹配服务 + 房间落某节点 | 本文 §4.2、§6.5 |

**实现顺序建议**：先单节点多 bridge 跑通（[game_server_gate_single_node.md](game_server_gate_single_node.md) §8.2），再按需加多节点与「仅本节点匹配」（本文 §3、§4.1）；**若需某地区全区服全玩家共同匹配**，优先采用「一地区一 Moon 节点」（本文 §6.3）；若该地区**战斗负载远大于匹配**且希望多机只扩房间，可采用「Center + Room」分离（本文 §6.6）：游戏服直连 Center 匹配、直连 Room 房间，**无需 Gate**。最后若需要跨区或地区内多节点，再上全局/地区匹配与房间落节点（本文 §4.2、§6.5）。
